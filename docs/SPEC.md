# IOC System — เอกสารโปรเจกต

ระบบบริหารจัดการคำร้องภายในองค์กร (Internal Ticket Management System) ที่ออกแบบมาเพื่อเพิ่มประสิทธิภาพในการติดตามงานและการสื่อสารระหว่างแผนก

## Tech Stack

* **Frontend/Backend:** Next.js
* **Database Management:** Prisma ORM
* **Database:** MySQL (Hosted on Plesk)
* **Icons:** Lucide Icons
* **Visualization:** Impeccable Graphify

---

## สิทธิ์การใช้งาน (User Roles & Permissions)

### 1. พนักงาน (Staff)

* เข้าสู่ระบบและออกจากระบบ (Login / Logout)
* รีเซ็ตรหัสผ่าน (Reset Password)
* ดูรายการคำร้องของตนเอง (View Ticket List)
* สร้างคำร้องใหม่พร้อมแนบไฟล์ (Create Ticket & Attach Files)
* จัดการความคิดเห็น (Create, Edit, Delete Comment)
* ติดตามสถานะงาน (Track Status)
* แก้ไขข้อมูลคำร้อง (เฉพาะก่อนเจ้าหน้าที่รับงาน)
* ยกเลิกคำร้อง (ไม่อนุญาตหากสถานะเป็น "เสร็จสมบูรณ์")
* แก้ไขและส่งคำร้องใหม่กรณีถูกปฏิเสธ (Resubmit after Reject)

### 2. เจ้าหน้าที่ (Officer)

* รับเรื่องเพื่อตรวจสอบ (Acknowledge Ticket — ยังไม่เริ่มดำเนินการจริง)
* บันทึกผลประเมิน (Save Evaluation — ก่อนส่งอนุมัติ)
* ส่งเรื่องขออนุมัติ (Submit for Approval)
* ส่งมอบ/ปิดงานเมื่อดำเนินการเสร็จ (Complete Ticket with summary)
* ปรับปรุงสถานะงาน (Update Status)
* มอบหมายงาน (Assign Ticket)
* รายงานความคืบหน้า (Update Progress)
* บริหารจัดการลำดับขั้นตอนงาน (Workflow Management)
* มอบหมายงานข้ามแผนกตามโครงสร้างองค์กร (Cross-department Assignment)
* ดูคำร้องทั้งหมดในแผนก/ที่เกี่ยวข้องได้ (หน้า **คำร้องทั้งหมด** — อ่านอย่างเดียว ไม่มีปุ่มดำเนินการ)
* ดำเนินการ workflow ได้จาก **กล่องงาน** (งานรอรับ + งานของฉัน) เท่านั้น

### 3. ผู้จัดการ (Manager)

* อนุมัติให้ดำเนินการ (Approve to Proceed — เปลี่ยนเป็นกำลังดำเนินการ)
* ปฏิเสธคำร้องพร้อมระบุเหตุผล (Reject Ticket with Comment)
* ตรวจสอบประวัติการอนุมัติ (View Approval History)
* ตรวจสอบภาพรวมผ่าน Dashboard (View Dashboard)

### 4. ผู้ดูแลระบบ (Admin)

* บริหารจัดการผู้ใช้งานและสิทธิ์การเข้าถึง (User & Role Management)
* รีเซ็ตรหัสผ่านให้ผู้ใช้ (Admin Reset Password — ยกเว้นบัญชีแอดมินและบัญชีตนเอง)
* บริหารจัดการข้อมูลแผนก (Department Management)
* ตรวจสอบบันทึกกิจกรรมย้อนหลัง (View Audit Logs)
* จัดการการลบข้อมูลแบบชั่วคราว (Soft Delete)

## ข้อมูลและระบบการทำงาน (System Specifications)

### ระบบยืนยันตัวตน (Authentication)

* ยืนยันตัวตนด้วย Username และ Password
* ระบบรีเซ็ตรหัสผ่าน
* บริหารจัดการ Session ด้วย JWT และตรวจสอบการหมดอายุ (Session Expiration)

### ข้อกำหนดของคำร้อง (Ticket Specifications)

* **ข้อมูลพื้นฐาน:** หัวข้อ (Title), รายละเอียด (Description), ระดับความสำคัญ (Priority), แผนกรับผิดชอบ (Department), วันที่เวลาเริ่มต้น (Scheduled Start), วันที่เวลาสิ้นสุด (Scheduled End), ไฟล์แนบ (Attachment)
* **เลขที่คำร้อง:** `{รหัสแผนกผู้ยื่น}-{YYYYMMDD}-{เลขรัน}` — รันต่อแผนกต่อวัน (โซน Asia/Bangkok) ไม่เติมศูนย์นำหน้า ขยายหลักอัตโนมัติ (1, 2, … 999, 1000) เช่น `IT-20260619-1`
* **ลำดับฟอร์มสร้างคำร้อง:** แผนกรับผิดชอบ → หัวข้อ → รายละเอียด → ความสำคัญ/ช่วงเวลา → ไฟล์แนบ (ฟอร์มเดียวทุกแผนก)
* **เงื่อนไขและกฎธุรกิจ:**
    * วันที่เวลาสิ้นสุดต้องอยู่หลังวันที่เวลาเริ่มต้น
    * ช่วงเวลาเริ่ม/สิ้นสุด **เลือกย้อนหลังได้** (แจ้งเรื่องที่เกิดไปแล้ว) — วันที่สร้างในระบบ (`createdAt`) บันทึกอัตโนมัติตอนกดส่ง แก้ไขไม่ได้
    * ขนาดไฟล์แนบรวมไม่เกิน 200 MB
    * แก้ไขคำร้องได้เฉพาะ "ก่อน" เจ้าหน้าที่รับเรื่องเพื่อตรวจสอบ
    * ห้ามแก้ไขหากมีผู้รับเรื่องแล้ว และห้ามยกเลิกหากสถานะ "เสร็จสมบูรณ์"
    * คำร้องที่ถูกปฏิเสธ (Reject) สามารถส่งใหม่ (Resubmit) ได้ไม่เกิน **3 ครั้ง** ต่อใบ — ถูกปฏิเสธครั้งที่ 4 ต้องสร้างคำร้องใหม่

### ลำดับ Workflow คำร้อง (Ticket Workflow)

**Stepper 5 ขั้น (UI):** คำร้องใหม่ → กำลังประเมิน → รออนุมัติ → กำลังดำเนินการ → เสร็จสิ้น  
(ขั้น «กำลังประเมิน» ใช้สถานะ DB `รอรับเรื่อง` + `receivedById` — ไม่เพิ่มสถานะใหม่)

ลำดับสถานะหลักใน DB:

1. **รอรับเรื่อง** — พนักงานสร้างคำร้อง (หลังรับเรื่อง = ขั้นประเมิน)
2. **รออนุมัติ** — เจ้าหน้าที่บันทึกผลประเมินแล้วส่งให้ผู้จัดการอนุมัติ **ก่อน** ดำเนินการจริง
3. **กำลังดำเนินการ** — ผู้จัดการอนุมัติแล้ว เจ้าหน้าที่ดำเนินการตามคำร้อง
4. **เสร็จสมบูรณ์** — เจ้าหน้าที่ส่งมอบ/ปิดงานเมื่อเสร็จ

#### ผลประเมิน (Ticket Evaluation)

เจ้าหน้าที่บันทึกหลังตรวจสอบจริง **ก่อน** ส่งอนุมัติ:

| ฟิลด์ | บังคับ | ตัวอย่าง |
|-------|--------|----------|
| ผลการตรวจสอบ | ใช่ | แรมชำรุด ไม่ boot |
| ประมาณค่าใช้จ่าย | ไม่ | ติ๊ก «มีค่าใช้จ่าย» แล้วกรอกจำนวนได้ถ้าทราบ — ไม่บังคับ |
| หมายเหตุ | ไม่ | มีสต็อกอะไหล่ |

ฟอร์มประเมินมาตรฐานเดียวทุกคำร้อง — ไม่มีฟิลด์ตามหมวด

<!-- เดิม: หมวดซ่อม/บำรุง (หลายเครื่อง) กรอกผลประเมินแยกต่อเครื่อง — ปิดใช้งานตามฟอร์มรวม -->

ผู้จัดการและพนักงานเห็นผลประเมินบนหน้ารายละเอียดคำร้อง

**กฎเปลี่ยนสถานะ:**

| จาก | ไป | ผู้ทำ |
|-----|-----|-------|
| รอรับเรื่อง | รอรับเรื่อง (บันทึกรับเรื่อง) | Officer — ตั้งผู้รับผิดชอบ ยังไม่เริ่มงาน |
| รอรับเรื่อง | รออนุมัติ | Officer — หลังบันทึกผลประเมินแล้ว |
| รออนุมัติ | กำลังดำเนินการ | Manager — อนุมัติให้ดำเนินการ |
| รออนุมัติ | ปฏิเสธ | Manager — ปฏิเสธพร้อมเหตุผล |
| กำลังดำเนินการ | เสร็จสมบูรณ์ | Officer — ส่งมอบ/ปิดงานพร้อมสรุป |
| ปฏิเสธ | รอรับเรื่อง | Staff — ส่งคำร้องใหม่ (Resubmit) |

**ห้าม** ข้ามจาก รอรับเรื่อง → กำลังดำเนินการ โดยตรง และ **ห้าม** ใช้การอนุมัติของ Manager เพื่อปิดงาน (เสร็จสมบูรณ์) ทันที

#### บันทึกการเปลี่ยน Workflow (2026-06)

| เปลี่ยนอะไร | เหตุผล |
|-------------|--------|
| แยก "รับเรื่อง" กับ "เริ่มดำเนินการ" | เดิมรับงานแล้วเข้า `กำลังดำเนินการ` ทันที ทำให้เอาทรัพย์สิน/เครื่องไปซ่อมได้ก่อนอนุมัติ |
| ย้าย `รออนุมัติ` มาก่อน `กำลังดำเนินการ` | องค์กรต้องอนุมัติก่อนดำเนินการที่มีค่าใช้จ่ายหรือส่งซ่อมนอก |
| Manager อนุมัติ → `กำลังดำเนินการ` (ไม่ใช่ `เสร็จสมบูรณ์`) | อนุมัติ = อนุญาตให้ทำต่อ ไม่ใช่ปิดงาน — Officer ปิดงานเมื่อซ่อม/ดำเนินการเสร็จ |
| Staff แก้ไขได้จนกว่า Officer จะรับเรื่อง | `receivedById` ถือว่ารับเรื่องแล้ว แม้สถานะยังเป็น `รอรับเรื่อง` |
| Stepper 5 ขั้น + ฟอร์มผลประเมิน | แยกขั้นประเมินให้เห็นชัด หัวหน้าอ่านสรุปก่อนอนุมัติ |

### การแจ้งเตือนในแอป (In-app Notifications)

* แจ้งเตือนผ่าน **กระดิ่ง** มุมขวาบน — รายการต่อ user, อ่านแล้ว/ยังไม่อ่าน (`readAt`)
* เก็บใน MySQL ตาราง `notifications` — ดึงเมื่อเปิดกระดิ่ง, รีเฟรชหน้า, หลัง action, กลับมาเปิดแท็บ, หรือ **SSE push** (`/api/realtime`) เมื่อมีเหตุการณ์ใหม่
* SSE แจ้ง client ให้ refetch กระดิ่งและรายการคำร้องทันที — poll ทุก 60 วิเป็น fallback เมื่อ connection หลุด
* คลิกรายการ → ไปหน้าที่เกี่ยวข้อง (ลิงก์ตาม role ของผู้รับ) และ mark อ่านแล้ว

### ฟีเจอร์ระดับองค์กร (Enterprise Features)

* **Search & Filter:** ระบบค้นหาและกรองข้อมูลขั้นสูง
* **Pagination:** การแบ่งหน้าข้อมูลเพื่อประสิทธิภาพ
* **Soft Delete:** ระบบลบข้อมูลแบบคงสถานะในฐานข้อมูล
* **Audit Logs:** บันทึกประวัติการเปลี่ยนแปลงข้อมูลสำคัญ
* **Error Handling & Rate Limiting:** ระบบจัดการข้อผิดพลาดและความปลอดภัย

## ข้อกำหนดในการพัฒนา (Development Guidelines)

1. **Strict Scope:** พัฒนาฟีเจอร์ตามที่ระบุในเอกสารนี้เท่านั้น
2. **Database:** ใช้ Prisma Migration ในการจัดการโครงสร้างฐานข้อมูลเสมอ
3. **Security:** ทุก API Endpoint ต้องผ่านการตรวจสอบสิทธิ์ (AuthN & AuthZ)
4. **Data Integrity:** ข้อมูลสำคัญต้องมี Audit Log บันทึกความเปลี่ยนแปลง
5. **Ambiguity:** หากความต้องการไม่ชัดเจน ให้ทำเครื่องหมาย TODO ไว้ ห้ามคาดเดาเอง

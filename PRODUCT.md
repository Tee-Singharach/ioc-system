# Product

## Register

product

## Users

พนักงาน (Staff), เจ้าหน้าที่ (Officer), ผู้จัดการ (Manager), และผู้ดูแลระบบ (Admin) ภายในองค์กรเดียวกัน

บริบทการใช้งาน: ทำงานประจำวัน เปิดระบบเพื่อยื่นคำร้อง ติดตามสถานะ มอบหมายงาน อนุมัติ หรือดูแลผู้ใช้ — ไม่ใช่ลูกค้าภายนอก

งานหลักต่อหน้าจอ: ดูรายการคำร้อง กรอง/ค้นหา สร้างหรือแก้ไขคำร้อง ดูรายละเอียดและความคิดเห็น ดำเนินการตาม role (รับงาน อนุมัติ ฯลฯ)

## Product Purpose

IOC System คือระบบบริหารจัดการคำร้องภายในองค์กร (Internal Ticket Management) เพื่อเพิ่มประสิทธิภาพการติดตามงานและการสื่อสารระหว่างแผนก

ความสำเร็จ = ผู้ใช้หาคำร้องได้เร็ว เข้าใจสถานะชัด ทำงานต่อได้โดยไม่สับสน และ UI สนับสนุน workflow ตาม role ตาม `docs/SPEC.md`

## Brand Personality

เป็นมิตร ทันสมัย อ่านง่าย

- ภาษา UI เป็นภาษาไทยเสมอ (label, ปุ่ม, ข้อความระบบ)
- โทนอบอุ่นแต่เป็นมืออาชีพ — internal tool ที่ไม่น่ากลัว ไม่เย็นจนเกินไป
- ความชัดเจนสำคัญกว่าความโชว์: สถานะ ความสำคัญ แผนก ต้องสแกนได้ทันที

## Anti-references

- หน้าตา AI slop: gradient ม่วง, glassmorphism, glow ฟุ่มเฟือย
- Landing-page marketing (hero ใหญ่, scroll storytelling) — นี่คือ product UI ไม่ใช่ brand site
- Copy เว่อร์ภาษาอังกฤษในจุดที่ผู้ใช้ไทยอ่านทุกวัน
- Dark mode (โปรเจกตกำหนด light theme เท่านั้น)
- Component คนละสไตล์ระหว่างหน้า (ปุ่ม/ฟอร์มไม่สอดคล้อง)

## Design Principles

1. **งานก่อนหน้าตา** — ทุกหน้าจอต้องช่วย task หลักของ role นั้นให้เสร็จเร็วขึ้น
2. **สถานะอ่านได้ในพริบตา** — badge, สี semantic, ลำดับข้อมูลในตารางต้องชัด
3. **ความสม่ำเสมอ** — ปุ่ม ฟอร์ม ตาราง ใช้ vocabulary เดียวกันทั้งแอป (อ้าง `DESIGN.md`)
4. **เรียบง่าย น่าเชื่อถือ** — density พอดี ไม่แน่นจนอึดอัด ไม่โล่งจนเสียบริบท
5. **ตาม SPEC เท่านั้น** — ไม่เพิ่มฟีเจอร์นอก spec; จุดไม่ชัดใส่ TODO

## Accessibility & Inclusion

- เป้าหมาย **WCAG 2.1 AA**
- ภาษา HTML `lang="th"`; label ฟอร์มและปุ่มต้องมีข้อความที่เข้าใจได้
- สีสถานะ (priority, status) ต้องไม่พึ่งสีอย่างเดียว — มีข้อความประกอบ
- รองรับ keyboard focus ที่มองเห็นได้; ลด motion ที่ไม่จำเป็น (respect `prefers-reduced-motion` เมื่อเพิ่ม animation)
